# Scenario: "Buenos Aires": Kubernetes Pod Crashing

**Level:** Medium  
**Tags:** `kubernetes`, `k8s`, `rbac`, `security`, `troubleshooting`  

## 📝 Problem Description
A deployment contains two pods inside the default namespace: `logger` and `logshipper`. The `logshipper` pod is stuck in a continuous `CrashLoopBackOff` loop. 

Initial diagnostics reveal the application crashes because it is completely forbidden from accessing or reading the standard output log streams generated by the sibling `logger` pod. Per scenario limitations, modifications to the immutable pod specification manifest are strictly prohibited.

## 💡 The Solution Strategy
When a containerized pod tries to programmatically interface with the Kubernetes API control plane (e.g., using a client library to read logs), it relies on an internal cryptographic token injected into its file structure. This identity corresponds to a cluster `ServiceAccount`. If that account lacks a proper Role-Based Access Control (RBAC) policy, the API server terminates the handshake with an HTTP `403 Forbidden` exception.

To resolve this without rewriting the deployment spec, we identify the associated `ServiceAccount` and provision an administrative `ClusterRoleBinding` live, granting the account the cluster authorization required to access pod log subsystems.

### The Commands
```bash
# 1. Inspect the historical logs of the failing pod instance to identify the crash footprint
sudo kubectl logs -l app=logshipper --previous
# Error signature: read_namespaced_pod_log: (403) Reason: Forbidden

# 2. Extract the exact ServiceAccount assigned to the active deployment namespace context
sudo kubectl get pod -l app=logshipper -o jsonpath='{.items[0].spec.serviceAccountName}'
# Output exposes the target identifier: logshipper-sa

# 3. Bind the isolated ServiceAccount to an authorized cluster blueprint (cluster-admin)
sudo kubectl create clusterrolebinding logshipper-api-patch \
  --clusterrole=cluster-admin \
  --serviceaccount=default:logshipper-sa

# 4. Stream cluster entity cycles to watch the container stabilize into service availability
sudo kubectl get pods -w
# Output transitions from CrashLoopBackOff into 1/1 Running / Ready state
